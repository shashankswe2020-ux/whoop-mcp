## Important Issues

### 1. "No NaN propagation" claim is incomplete — NaN in input propagates to output

- **File:** `src/tools/stats-utils.ts:5`
- **Problem:** The module header comment states "No NaN propagation — edge cases return defined values." This is true for *internal* edge cases: the `ssTot === 0` guard in `linearRegression` (line 138) and the `stdDev === 0` guard in `detectAnomalies` (line 170) prevent NaN from being generated by division-by-zero. However, none of the 6 functions validate that input values are finite numbers. If any function receives `NaN` in its input array (e.g., `mean([1, NaN, 3])`), the NaN propagates through arithmetic and appears in the return value.

  **Practical risk:** Low today — these functions are consumed only by analytical tools that get data from the WHOOP API, which returns valid numbers. But the header claim is a documented invariant that doesn't hold for all inputs.

- **Fix (option A — validate inputs, recommended):**
  ```typescript
  function assertNonEmpty(values: number[], name: string): void {
    if (values.length === 0) {
      throw new Error(`${name}: cannot operate on an empty array`);
    }
    if (values.some((v) => !Number.isFinite(v))) {
      throw new Error(`${name}: array contains non-finite values (NaN or Infinity)`);
    }
  }
  ```
  This makes the claim true for all callers and fails loudly on bad input.

- **Fix (option B — clarify docs):**
  ```typescript
  * No NaN propagation from valid (finite) inputs — internal edge cases return defined values.
  ```
  This narrows the claim to match actual behavior.

## Suggestions

### 1. `detectAnomalies` does not validate `threshold` parameter

- **File:** `src/tools/stats-utils.ts:160`
- A negative or zero `threshold` produces unintuitive results: threshold ≤ 0 flags all values that differ from the mean (including negligible differences) as anomalies, since `deviation > 0` is true for every non-mean value. While the default (2) is sane and internal callers will use reasonable values, a guard like `if (threshold <= 0) throw new Error(...)` would make the contract explicit and prevent surprises from future callers.

### 2. `trendDirection` docstring mentions three R² tiers but implementation uses two

- **File:** `src/tools/stats-utils.ts:186-189`
- The docstring says "R² thresholds" and lists three categories (≤ 0.4, > 0.4), but the spec mentions three tiers: "high > 0.7, medium > 0.4, low ≤ 0.4". The implementation correctly collapses high and medium since both use the slope sign — the behavior is identical. However, the docstring doesn't mention the unused high/medium distinction, creating a subtle mismatch with the spec. Either align the docstring with the spec (noting that high and medium produce the same behavior) or document the simplification decision.

### 3. No tests for NaN or Infinity input values

- **File:** `tests/tools/stats-utils.test.ts`
- The existing NaN-safety test (line 152) verifies that `linearRegression([5, 5, 5])` doesn't produce NaN — this tests the *internal* guard, not NaN *input*. Consider adding tests that document what happens with NaN or Infinity inputs, even if the decision is to let them propagate. This freezes the behavior and prevents surprise regressions:
  ```typescript
  it("propagates NaN from NaN input (known limitation)", () => {
    expect(Number.isNaN(mean([1, NaN, 3]))).toBe(true);
  });
  ```
  If option A from Important Issue #1 is adopted, these become rejection tests instead.

## What's Done Well

- **Mathematical correctness:** Population std dev (÷N, not ÷(N-1)) is the right choice for descriptive stats on WHOOP data. OLS regression uses closed-form sums for x-indices, avoiding unnecessary loops. R² computation via SSres/SStot is textbook.
- **NaN guard discipline:** The two critical NaN-generation points — `ssTot === 0` in `linearRegression` and `stdDev === 0` in `detectAnomalies` — are both explicitly guarded with clear comments explaining why.
- **Non-mutation:** `median` copies the array before sorting. The test on line 73 explicitly verifies this.
- **Defensive denominator guard:** The `denominator === 0` check in `linearRegression` (line 116) is mathematically impossible for x=0..n-1 with n≥2, but the guard is added anyway with a comment explaining the reasoning. Excellent defensive practice.
- **Test quality:** 37 tests with strong boundary coverage — constant data, single values, two values, unsorted input, and exact R² boundary thresholds (0.4 vs 0.41). The monotonic → R²≈1 and constant → R²=0 property-based tests are exactly what the spec requested.
- **Zero runtime dependencies:** Pure TypeScript, no external math library. Appropriate for the scale of these calculations (max 500 data points).
